Iteration 1

Vision Statement

Our project aims to bring the users’ dining experience to the next level. On the customer product, users will easily be able to check wait times, place reservations, make orders, and send payment from their phone. On the restaurant’s product, employees will be able to check the number of people in line, see orders that have been placed, receive payment, and see what tables are in need of service.


Features:

  • General
    • Login to account with facebook or email. Two different accounts, consumer and restaurant.
  • Consumer
    • Add themselves to waitlists at restaurants and receive notifications about position in queue and estimated wait time.
    • Order restaurant menu items through phone device.
    • Once in restaurant, diners will be able to ping servers for service instead of waiting for the server to arrive.
    • Extended Feature: Pay for order through phone
    • Extended Feature: Rate individual items on the menu
  • Restaurant
    • Maintain a waitlist
    • Make their own menu
    • Maintain queue of orders
    • See what tables are in need of service.
    • Extended feature: Receive payment from users’ phones.
    • Extended feature: Have a section for specialty items

Actors:

  • Diner
    • Add to waitlist
    • Place orders
    • Make payments
    • Call for server
  • Restaurant Staff
    • Keep track of diners and waitlist
    • Make, cancel waitlist positions
    • View, make, cancel orders
    • Approve, cancel payments
  • Manager (Admin)
    • Modify menu items
    • Approve/boot staff
    • Set staff active times?
    • Check restaurant status

Use Cases

Title: Customers putting their names down for the restaurant
Primary Actor: Customer
Secondary Actors: Management
Preconditions: Management has entered in the approximate wait time per table.
Main Path:

  1. User opens the app.
  2. User clicks the button, “Add name to list”.
  3. User can see their number in line and the approximate wait time. This changes dynamically in accordance to people canceling their place in line or getting seated.

Title: Management viewing and modifying waitlist.
Primary Actor: Management
Secondary Actors: None
Preconditions: None
Main Path:

  1. User opens the app/web browser.
  2. User can see the number of people in line.
  3. User can add customers, cancel placements and modify placements.

Title: Management viewing the number of people in line and seating them.
Primary Actor: Management
Secondary Actors: Customers
Preconditions: None
Main Path:

  1. User opens the app/web browser.
  2. User can view the name and contact information of each person in line.

Title: Customer placing an order.
Primary Actor: Customer
Secondary Actors: Management, Staff
Preconditions: Management has entered in the restaurant’s menu. EITHER: management has linked the user with a table number or the user is on the current wait queue.
Main Path:

  1. User opens the app.
  2. User adds their orders to a cart.
  3. User selects, “Place Order”.
  4. IF the user is already seated, the user can see the approximate wait time for their food and make changes within 10 minutes. ELSE if the user is waiting in line, their order is saved.

Title: Staff receiving an order.
Primary Actor: Staff
Secondary Actors: Customer.
Preconditions: None.
Main Path:

  1. User opens the app/web browser.
  2. User can view the orders from each table in the restaurant.
  3. User can also view orders made by people currently waiting in line.
  4. Users can remove orders when they have been completed, and set the estimated wait time for completion.

Title: Customer calls for service
Primary Actor: Customer
Secondary Actors: Staff, Management
Preconditions: User has been seated.
Main Path:

  1. User opens the app.
  2. User can ping a server with the following types, “Order”, “Payment”, “Other”, and include a note, if necessary.

Title: Staff receives call for service.
Primary Actor: Staff
Secondary Actors: Customer
Preconditions: User has been seated and created a ping.
Main Path:

  1. If the staff has been assigned to the table that the customer is seated at, they will receive a notification with the user’s note.
  2. The staff can view a queue of all the pings that have been directed at them and can remove pings as they are dealt with.

Title: Customer makes payment.
Primary Actor: Customer
Preconditions: User has been seated and has made an order.
Main Path:

  1. If user has not done so already, user enters in their credit card information.
  2. User is shown their receipt and can approve their payment.
  3. User’s credit card is charged.

Proposed Architecture

Packages:

  • Restaurant:
    • Components:
      • Queue of reservations stored in the Parse database Ping customers when their reservation is available Organize customers and edit their info (Assign to tables) View information on customer orders and edit these orders as necessary
  • Menu:
    • Components:
      • OpenGl to implement the user interface, orders will be sent to the Restaurant.
  • Customer:
    • Components:
      • Reserve a spot at the Restaurant
      • Functionality to ping the restaurant
      • Contains a transition to the Menu package for ordering
      • Make payments for orders

Resources

  • Parse API for accessing facebook/email, having a functional database to store necessary information, and allowing for push notifications
  • OpenGl for interactive menu animations (Adding/removing desired ingredients)
  • Android for the device
  • Extended Features: Stripe/Venmo/Google Wallet for payment
